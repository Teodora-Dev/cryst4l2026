Your old ITCSS brain, but modern CSS.

Cryst4l is a pure-CSS rewrite of your original Sass starter, keeping the same ITCSS structure while using native @layer, custom properties (tokens), fluid scales, and optional container-query utilities.

ITCSS structure preserved @layer ordered cascade No Sass required Fluid type + spacing tokens CQ helpers mirror your suffix naming

Quick start

<link rel="stylesheet" href="/path/to/cryst4l/source/css/styles.css">

That’s it — styles.css imports each module into the correct layer using: @import url("...") layer(utilities);

Layers (ITCSS, now native)

Cryst4l uses a fixed layer order so you get predictable overrides without specificity wars. 

@layer reset, settings, base, objects, components, utilities;

What goes where

  • reset — minimal reset and sensible defaults
  • settings — tokens: colour, spacing, typography, breakpoints
  • base — element styling (type, links, forms)
  • objects — layout primitives (wrappers, grid)
  • components — UI components (kept as a placeholder layer)
  • utilities — single-purpose helpers with !important

Tokens (custom properties)

Tokens live in _1-settings/_variables.css. You can theme by overriding tokens — usually you don’t need new classes.

Fluid type scale

:root {
  --fs-0:  clamp(1rem, 0.94rem + 0.40vw, 1.12rem);
  --fs-3:  clamp(1.52rem, 1.30rem + 1.10vw, 2.05rem);
}

Utility wrappers for tokens

Optional helpers (utilities) you can use directly:

<h2 class="u-fs-3 u-lh-tight">A nice headline</h2>
<div class="u-flow">

  <p>Paragraph 2</p>
</div>

Grid

The grid keeps your familiar class API: o-grid, o-grid__item, and width fractions like u-1/2 (including responsive suffixes). 

<div class="o-grid">
  <div class="o-grid__item u-1/2@md-up">Half</div>
  <div class="o-grid__item u-1/2@md-up">Half</div>
</div>
u-1/2@md-up
u-1/2@md-up

Push/pull helpers are still there (u-push-*, u-pull-*) and remain position: relative-based offsets.

Utilities

Utilities are split into helpers under _7-utilities/helpers and imported via _utilities.css.

Common ones

  • Spacing: u-margin-*, u-padding-* (+ responsive variants)
  • Fractions: u-1/2, u-2/3, etc (+ responsive variants)
  • Quick tokens: u-gap-*, u-px-*, u-mb-* (uses --space-*)
  • Helpers: clearfix, hidden/show, colours, borders, buttons

Example

<div class="u-p-4 u-mb-4">
  <button class="u-button">Button helper</button>
</div>

Container queries (optional utilities)

Container queries let components respond to their own container width, not the viewport. Cryst4l includes CQ helpers that mirror your breakpoint suffix style. 

<div class="u-container o-grid">
  <div class="o-grid__item u-cq-1/2@md-up">Half when container ≥ md</div>
  <div class="o-grid__item u-cq-1/2@md-up">Half when container ≥ md</div>
</div>

What you get

  • .u-container + named containers (.u-container-card, etc)
  • Visibility: .u-show@md-up / .u-hide@lg-up (container-based)
  • Fractions: .u-cq-1/2@sm-up, .u-cq-2/3@lg-up, etc
  • Layout helper: .u-cq-row + .u-cq-row@md/@lg (stack → row)

Lean builds (recommended)

If you don’t need every helper, you can ship a smaller bundle by importing only what you use. Here are two practical approaches.

A) Utilities preset file

Create a file like utilities.build.css and import only the helpers you want:

/* utilities.build.css */
@layer utilities;

@import url("./_7-utilities/helpers/_widths.css") layer(utilities);
@import url("./_7-utilities/helpers/_margins.css") layer(utilities);
@import url("./_7-utilities/helpers/_padding.css") layer(utilities);
/* skip colours/borders/buttons if unused */

B) Minimal entrypoint

Make a lean entry file for production (still respects the same layer order): 

/* styles.build.css */
@layer reset, settings, base, objects, components, utilities;

@import url("./_3-resets/_normalize.css") layer(reset);
@import url("./_1-settings/_variables.css") layer(settings);
@import url("./_4-base/_base.css") layer(base);
@import url("./_5-objects/_wrappers.css") layer(objects);
@import url("./_5-objects/_grid.css") layer(objects);
/* import only selected utilities */

Rule of thumb

Start lean, add helpers as-needed. Keeping utilities modular is the whole point of the folder split.
