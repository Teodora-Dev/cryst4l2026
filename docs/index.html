01-introduction.md

Cryst4l is a **foundation framework**, not a design system.

It gives you:

  • a predictable cascade
  • layout primitives
  • neutral structural components
  • a utility layer for explicit overrides
  • tokens for theming and scaling

It does **not** give you:

  • pre-styled components
  • a “framework look”
  • a theme engine

If you want quick visual polish, add a design system on top. If you want CSS that stays sane for years, start here.

02-architecture.md

Cryst4l follows **ITCSS**, implemented natively via CSS @layer.


@layer reset, settings, base, objects, components, utilities;

Each layer has:

  • a single responsibility
  • strict rules
  • a defined place in the cascade

Result: fewer specificity hacks, fewer “where is this coming from?” moments.

03-layers.md

Cryst4l ships six layers (in order):

Reset

Browser normalisation only. No classes. No opinions.

Settings

Design tokens only (CSS custom properties).

Base

Element-level defaults (body, a, input, etc.).

Objects

Layout primitives (o-wrapper, o-grid).

Components

Semantic structure without forced styling (c-*).

Utilities

Single-purpose overrides (u-*). Allowed to use !important.

04-objects.md

Objects are **layout-only** primitives: reusable, context-free, and structural.

Included objects:

  • o-wrapper
  • o-grid
  • o-grid__item

Example:


<div class="o-grid">
  <div class="o-grid__item u-1/2@md-up">…</div>
  <div class="o-grid__item u-1/2@md-up">…</div>
</div>

Rules:

  • no colours
  • no typography
  • no theme assumptions

05-components.md

Components are **contracts**, not designs.

They define:

  • structure
  • relationships
  • behaviour hooks

Core components included:

  • c-box
  • c-stack (+ c-stack--tight, c-stack--loose)
  • c-cluster
  • c-card
  • c-media
  • c-switcher
  • c-button
  • c-list
  • c-disclosure (for <details>)

Example:


<article class="c-card u-border u-p-4">
  <header class="c-cluster">
    <h2 class="u-fs-2">Title</h2>
    <span class="u-fs--1">Meta</span>
  </header>

  <div class="c-stack">
    <p>Paragraph one</p>
    <p>Paragraph two</p>
  </div>
</article>

All visual decisions live in **utilities** or **tokens**.

06-utilities.md

Utilities are **explicit overrides**: local, single-purpose, and predictable.

Cryst4l supports two tiers.

Tier 1 — Classic responsive utilities

Viewport-based helpers (ported from the original Sass system).

Examples:

  • u-1/2@md-up
  • u-margin-bottom@lg-up
  • u-push-1/3@sm-up

Use when:

  • layout must respond to the viewport
  • you need precise breakpoint control

Tier 2 — Token utilities

Modern, small, token-backed shortcuts (2026 upgrade).

Examples:

  • u-p-4, u-mb-3, u-gap-2
  • u-fs-2, u-lh-tight, u-flow

Use when:

  • you want fast spacing/type tweaks
  • you want fluid scaling “for free”

You can use either tier or both.

07-tokens-and-theming.md

Tokens live in:

  • source/css/_1-settings/_variables.css (framework source)
  • docs/css/_1-settings/_variables.css (docs copy)

They define:

  • spacing (--space-*)
  • typography (--fs-*)
  • optional container breakpoint values (--cq-*, documented)
  • any brand/theme tokens you add (e.g. --surface-bg)

Theming

Theme Cryst4l by overriding tokens — not by rewriting components:


[data-theme="dark"]{
  --surface-bg: #111;
  --text-color: #eee;
}

This keeps your components portable and your CSS small.

08-queries.md

Media queries

Use literal values:


@media (min-width: 720px){ … }

CSS custom properties **cannot** be used inside media query conditions.

Container queries (recommended)

Container queries let components respond to their container, not the viewport.

Enable containers with:


<div class="u-container">…</div>

Then use CQ utilities like:


<div class="u-container o-grid">
  <div class="o-grid__item u-cq-1/2@md-up">…</div>
  <div class="o-grid__item u-cq-1/2@md-up">…</div>
</div>

Cryst4l’s CQ utilities mirror the @*-up suffix convention.

09-composition.md

Cryst4l is designed for **composition**.

A common pattern:

  • c-* provides structure
  • u-* provides visual decisions
  • tokens provide scaling and theming

Example:


<article class="c-card u-border u-p-4">
  <header class="c-cluster u-mb-3">
    <h2 class="u-fs-2">Title</h2>
    <span class="u-fs--1">Meta</span>
  </header>

  <div class="c-stack">
    <p>Paragraph one</p>
    <p>Paragraph two</p>
  </div>

  <footer class="c-cluster u-mt-4">
    <button class="c-button u-px-3 u-py-2 u-border">Action</button>
  </footer>
</article>

10-lean-builds.md

Cryst4l is modular by design: ship only what you use.

A) Lean entry file

Create styles.build.css:


@layer reset, settings, base, objects, components, utilities;

@import url("./_3-resets/_normalize.css") layer(reset);
@import url("./_1-settings/_variables.css") layer(settings);
@import url("./_4-base/_base.css") layer(base);
@import url("./_5-objects/_wrappers.css") layer(objects);
@import url("./_5-objects/_grid.css") layer(objects);
@import url("./_6-components/_components-core.css") layer(components);

/* import only the utility helpers you use */

B) Pick utilities à la carte

Instead of importing _utilities.css, import just helpers:


@import url("./_7-utilities/helpers/_widths.css") layer(utilities);
@import url("./_7-utilities/helpers/_margins.css") layer(utilities);
@import url("./_7-utilities/helpers/_padding.css") layer(utilities);

11-philosophy.md

Cryst4l follows one rule:

If you add styling to a component, stop.

If you add structure to a utility, stop.

  • Structure lives in **objects** and **components**
  • Style lives in **tokens** and **utilities**

This keeps your CSS predictable and your UI flexible.
